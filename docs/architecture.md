# Архитектура приложения

## 1. Общее описание

Архитектура проекта построена на стандартной структуре Django: маршруты определяются в **URL‑конфигурации**, логика страниц — во **views**, а данные обрабатываются через модели и формы. Основные пользовательские сценарии:
- авторизация
- регистрация
- управление бюджетом
- создание и удаление транзакций
- фильтрация и отчёты
## 2. Маршрутизация (URL Routing)

### 2.1. Главный маршрутизатор

Главный `urls.py` подключает приложение страниц и стандартные Django‑вью:

```
/
├── admin/
├── (pages.urls)
└── logout/
```

### 2.2. Маршруты приложения pages

| URL                          | View                        | Назначение          |
| ---------------------------- | --------------------------- | ------------------- |
| `/`                          | **login_view**              | Авторизация         |
| `/register/`                 | **register_view**           | Регистрация         |
| `/dashboard/`                | **dashboard_view**          | Личный кабинет      |
| `/transactions/delete/<id>/` | **delete_transaction_view** | Удаление транзакции |
## 3. Архитектура Views

### 3.1. login_view

**Назначение:** обработка входа пользователя.

**Основная логика:**
- принимает POST‑данные
- вызывает `authenticate()`
- при успешной авторизации выполняет `login()` и перенаправляет на Dashboard
- при ошибке выводит сообщение через `messages.error`
- GET → отображение формы входа

**Шаблон:** `pages/login.html`
### 3.2. register_view

**Назначение:** регистрация нового пользователя.

**Основная логика:**
- проверка совпадения паролей
- проверка уникальности имени пользователя
- создание пользователя через `create_user()`
- перенаправление на страницу входа

**Шаблон:** `pages/register.html`
### 3.3. dashboard_view

**Главная страница приложения.**

**Функциональность:**
- отображение бюджета
- изменение месячного лимита
- добавление транзакций
- фильтрация по датам
- отчёты по категориям
- расчёт остатка бюджета
- список транзакций

**Основная логика:**
#### 1) Получение бюджета пользователя

```
budget, _ = Budget.objects.get_or_create(user=request.user)
```

#### 2) POST‑обработка

- если в POST есть `monthly_limit` → обновление бюджета
- иначе → создание транзакции через `TransactionForm`
#### 3) Фильтрация периода

```
?start=YYYY-MM-DD&end=YYYY-MM-DD
```
#### 4) Отчёты

- расходы по категориям
- доходы по категориям
#### 5) Расчёт остатка бюджета

```
remaining = budget.monthly_limit - monthly_expenses
```

**Шаблон:** `pages/dashboard.html`
### 3.4. delete_transaction_view

**Назначение:** удаление транзакции, принадлежащей текущему пользователю.

**Логика:**

```
Transaction.objects.filter(id=transaction_id, user=request.user).delete()
```

**Безопасность:** Пользователь может удалить **только свои** транзакции.

## 4. Пользовательский поток (User Flow)

### 4.1. Вход

Пользователь открывает `/`, вводит данные и попадает в Dashboard.
### 4.2. Dashboard

Пользователь может:
- изменить бюджет
- добавить транзакцию
- фильтровать период
- просматривать отчёты
- удалять транзакции
### 4.3. Выход

`/logout/` → LogoutView → возврат на `/`
## 5. Диаграмма взаимодействия

```
DashboardView
│
├── BudgetForm → Budget
├── TransactionForm → Transaction
├── Category.objects.all()
└── Отчёты (annotate, aggregate)
```

## 7. Итог

Архитектура приложения опирается на три ключевых компонента:
- **Views** — бизнес‑логика
- **Templates** — отображение
- **Models** — данные

Центральная точка взаимодействия — Dashboard, объединяющий бюджет, транзакции и отчёты.
